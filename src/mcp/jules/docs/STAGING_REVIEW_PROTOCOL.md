# ระบบตรวจสอบความจำ (Staging Review Protocol)
<!-- PRIORITY:0 -->
> **สถานะ:** ร่างมาตรฐาน (Draft)
> **เป้าหมาย:** เพื่อให้อาจารย์เป็นผู้ตัดสินใจขั้นสุดท้ายก่อนมีการ "ลบ" หรือ "ยุบรวม" ข้อมูลใดๆ

เพื่อให้ "แรงงาน AI" ไม่เผลอลบข้อมูลสำคัญ ระบบ Memory Manager จะต้องผ่านขั้นตอน **Staging Review** เสมอ โดยมีขั้นตอนดังนี้:

---

## 📝 1. การสร้างไฟล์ `MEMORY_STAGING.md`
เมื่อระบบพบว่า Context เริ่มล้น หรือความจำเก่าเกิน 30 วัน ระบบจะสร้างไฟล์รายงานใน Root ของโปรเจกต์:
- **Proposed Changes:** รายการข้อมูลที่ต้องการลบ (Prune) หรือ ยุบรวม (Summarize)
- **Rationale:** เหตุผลว่าทำไมถึงต้องจัดการ (เช่น "ข้อมูลชุดนี้เป็น Priority 3 และไม่ได้ถูกเรียกใช้นานเกิน 1 เดือน")
- **Risk Assessment:** ผลกระทบที่อาจเกิดขึ้น (เช่น "อาจทำให้ลืมรายละเอียดของเซสชันที่ 5 ไปบ้าง")

## 🛡️ 2. รูปแบบการตัดสินใจ (The Decision Gate)
อาจารย์สามารถตรวจงานและสั่งการได้ง่ายๆ โดยการแก้ไขไฟล์ หรือบอกผมในแชท:
- **[ ] Approve All:** อนุญาตให้ทำตามแผนทั้งหมด
- **[ ] Reject All:** ห้ามลบข้อมูลใดๆ และเก็บไว้เหมือนเดิม
- **[ ] Partial Approve:** อนุญาตเฉพาะบางรายการ

## 🛑 3. กฎความปลอดภัย (The Red Line)
- **No Manual Edit = No Action:** หากอาจารย์ไม่ตอบรับหรือไม่อนุมัติ ระบบห้ามทำการเปลี่ยนแปลงใดๆ กับไฟล์ต้นทางเด็ดขาด
- **Audit Backups:** ก่อนมีการจัดการความจำ ระบบต้องสร้างไฟล์ Backup (`.bak`) ของไฟล์เดิมไว้เสมอ

---

## 💡 ตัวอย่างไฟล์ `MEMORY_STAGING.md`
```markdown
# 📥 Memory Management Proposal (2024-04-24)

### 📋 รายการที่เสนอจัดการ
1. **[Summarize]** รายละเอียดการติดตั้ง Node.js ในเซสชันที่ 2
   - *เหตุผล:* ข้อมูลยาวเกินไป และตอนนี้ระบบรันได้ปกติแล้ว
2. **[Prune]** รายการบั๊กที่แก้เสร็จแล้วใน Track A
   - *เหตุผล:* งานจบแล้ว (Completed) และบันทึกไว้ใน Archive แล้ว

### 🚩 การตัดสินใจของอาจารย์
- [ ] อนุมัติทั้งหมด
- [ ] ปฏิเสธทั้งหมด
```
