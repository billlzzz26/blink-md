# มาตรฐานโครงสร้างไฟล์บริบท (Context File Protocol)
<!-- PRIORITY:0 -->
> **สถานะ:** ร่างมาตรฐาน (Draft)
> **ใช้กับ:** `GEMINI.md`, `AGENTS.md`, `CLAUDE.md`, `SOUL.md`

เพื่อให้ระบบจัดการความจำ (Memory Manager) ทำงานได้อย่างปลอดภัยและแม่นยำ ไฟล์บริบททุกไฟล์ต้องปฏิบัติตามโครงสร้างระดับความสำคัญ (Priority Levels) ดังนี้:

---

## 🛑 [LEVEL 0] IMMUTABLE RULES (กฎเหล็กถาวร)
- **นิยาม:** กฎพื้นฐานที่ห้ามเปลี่ยนแปลงโดยเด็ดขาด เพราะมีผลต่อความปลอดภัยและการสื่อสาร
- **ตัวอย่าง:** "ผู้ใช้เขียนโค้ดไม่เป็น", "สื่อสารเป็นภาษาไทยเสมอ", "ความปลอดภัยของ Credential"
- **การจัดการ:** **ห้ามลบ ห้ามแก้ไขอัตโนมัติ** (Read-Only for machines, Manual edit by user ONLY)

## 🏗️ [LEVEL 1] CORE WORKFLOWS (วงจรการทำงาน)
- **นิยาม:** มาตรฐานและกระบวนการทำงานที่ตกลงกันไว้
- **ตัวอย่าง:** "Conductor Protocol", "Interactive Memory Protocol", "TDD Pattern"
- **การจัดการ:** **ห้ามลบ** สามารถเพิ่มข้อมูลได้ แต่การแก้ไขเนื้อหาเดิมต้องทำด้วยมือเท่านั้น

## 📚 [LEVEL 2] ACCUMULATED KNOWLEDGE (ความรู้สะสม)
- **นิยาม:** ข้อมูลทางเทคนิคหรือรายละเอียดโปรเจกต์ที่สรุปมาแล้ว
- **ตัวอย่าง:** "ผลการวิจัย Hermes Agent", "โครงสร้างโฟลเดอร์โปรเจกต์"
- **การจัดการ:** สามารถ **สรุป (Summarize)** หรือ **ยุบรวม (Deduplicate)** ได้เมื่อข้อมูลล้น แต่ต้องผ่านการอนุมัติ

## ⏱️ [LEVEL 3] TRANSIENT MEMORY (ความจำชั่วคราว)
- **นิยาม:** ข้อมูลเฉพาะกิจหรือสถานะงานรายวัน
- **ตัวอย่าง:** "สิ่งที่ต้องทำในวันนี้", "รายการไฟล์ที่แก้ล่าสุด"
- **การจัดการ:** สามารถ **ลบ (Prune)** หรือ **Archive** ได้ตามรอบเวลา (เช่น ทุก 30 วัน)

---

## 🛡️ มาตรการความปลอดภัย (Guardrails)
1. **Machine-Readable Tags:** ใช้ HTML comments (เช่น `<!-- PRIORITY:0 -->`) เพื่อกำกับแต่ละส่วน
2. **Staging Review:** ทุกการเปลี่ยนแปลงอัตโนมัติ ต้องถูกบันทึกในไฟล์ `MEMORY_STAGING.md` และรอการยืนยันจากผู้ใช้
3. **Audit Trail:** ต้องระบุเสมอว่าความจำนี้ถูกเพิ่ม/แก้ไข เมื่อไหร่ และโดยใคร
